# 数据库表设计规范
这份规范**专门针对高并发、大数据量、低延迟要求**的业务场景（如电商、支付、社交、直播），遵循**性能优先、冗余可控、避免锁竞争、便于水平扩展**原则，是互联网大厂通用实战标准。

---

## 一、核心设计总原则（必须遵守）
1. **单表数据量控制**
   - 单表物理行不超过 **2000万**（InnoDB 最优阈值），超过必须**分库分表**
   - 单表字段数不超过 **30 个**，避免宽表
2. **索引绝对原则**
   - 单表索引总数 ≤ **6 个**
   - 联合索引字段数 ≤ **5 个**
   - 禁止在高更新字段建索引
3. **事务与锁控制**
   - 事务执行时间 < **50ms**，避免长事务
   - 禁止使用 `SELECT ... FOR UPDATE` 大范围锁
4. **字段设计**
   - 所有字段**必须 NOT NULL**，用默认值替代 NULL
   - 禁止使用 TEXT/BLOB 大字段，必须拆分到副表

---

## 二、库与表基础规范
### 1. 库名、表名命名规范
- 统一使用**小写英文 + 下划线**，禁止数字开头、禁止特殊字符
- 业务库：`业务名_功能名`，例：`mall_order`、`user_center`
- 表名：`业务模块_表名`，例：`order_info`、`user_account`
- 中间表：`表a_表b_relation`
- 历史表/归档表：`表名_hist`、`表名_202603`

### 2. 存储引擎与字符集
- **默认引擎：InnoDB**（唯一支持事务、行锁、外键、崩溃恢复）
- 禁止使用 MyISAM（无事务、表锁、高并发极易雪崩）
- 字符集：**utf8mb4**（支持 emoji，兼容所有字符）
- 排序规则：`utf8mb4_general_ci`（性能优先）/ `utf8mb4_unicode_ci`（精准排序）

### 3. 必须的公共字段（所有表强制）
```sql
`id`          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
`create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
`update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
`is_deleted`  TINYINT NOT NULL DEFAULT 0 COMMENT '删除标识 0=未删除 1=已删除',
PRIMARY KEY (`id`)
```
- 高并发必须**逻辑删除**，禁止物理 `DELETE`
- `update_time` 用于数据同步、缓存失效、分表路由

---

## 三、字段设计规范（高并发核心）
### 1. 主键设计（生死线）
- **类型：BIGINT（必选）**
  - 禁止 INT（20亿上限，高并发易溢出）
  - 禁止 VARCHAR（性能差、无法分表、无法索引优化）
- **主键生成策略**
  - 分布式场景：**雪花算法（Snowflake）** / 百度 UidGenerator / 美团 Leaf
  - 禁止业务字段做主键（如订单号、手机号）
  - 禁止多字段联合主键

### 2. 字段类型选择（性能从高到低）
**数字 > 日期 > 枚举 > 字符串**
1. **数值类型**
   - 状态/类型：`TINYINT`（-128~127）
   - 数量/金额：`INT` / `BIGINT`（金额统一用分存储，避免浮点数精度丢失）
   - 禁止：FLOAT、DOUBLE、DECIMAL（非金额场景）
2. **字符串类型**
   - 固定长度：`CHAR`（如手机号、身份证）
   - 可变长度：`VARCHAR`（长度按需，不要无脑 255）
   - 禁止：TEXT、BLOB（必须拆表）
3. **日期类型**
   - 统一：`DATETIME`（时区无关、范围大）
   - 禁止：TIMESTAMP（2038 年溢出问题）、字符串存时间

### 3. 字段约束强制要求
- 所有字段 **NOT NULL**，设置合理默认值
  - 字符串默认 `''`
  - 数字默认 `0`
  - 时间默认 `CURRENT_TIMESTAMP`
- 禁止 NULL：会导致索引失效、查询优化困难、业务异常

### 4. 禁止使用的 MySQL 特性
- 禁止外键约束（高并发导致锁等待、死锁，交给业务层控制）
- 禁止触发器、存储过程、函数（无法水平扩展、排错困难）
- 禁止视图（性能极差，影响主表查询）
- 禁止字段允许 NULL

---

## 四、索引设计规范（高并发性能关键）
### 1. 索引基础规则
- 索引名格式：`idx_字段名` / `idx_字段1_字段2`
- 唯一索引名：`uk_字段名`
- **最左前缀原则**：联合索引严格按查询顺序建立
- 高频查询字段必须建索引，低频/全表扫描禁止索引

### 2. 必须建索引的场景
- WHERE 条件中频繁使用的字段
- ORDER BY / GROUP BY 字段
- 多表 JOIN 关联字段
- 范围查询字段（>、<、BETWEEN）

### 3. 禁止建索引的场景
- 高更新频率字段（如库存、余额、状态）
- 区分度低的字段（如性别、状态仅 2-3 个值）
- 单表索引超过 6 个
- 模糊查询 `%xxx`（无法命中索引）

### 4. 联合索引设计技巧
```sql
-- 最优：where a=? and b=? order by c
INDEX idx_a_b_c (a,b,c)
```
- 等值查询放前面，范围查询放后面
- 区分度高的字段放前面

## 五、高并发优化设计规范
### 1. 大字段/冷字段拆分
- 原则：**查询频繁的字段做主表，大字段、低频字段做副表**
- 主表：`order_info`（核心字段）
- 副表：`order_detail`（描述、备注、扩展字段）

### 2. 冗余字段（空间换时间，高并发必用）
允许适度冗余，**避免跨表 JOIN**（JOIN 高并发性能极差）
例：订单表冗余用户名、用户头像，避免查询时关联用户表

### 3. 分库分表设计（单表超 2000 万必须）
- 分表维度：**用户 ID、订单 ID、商户 ID**
- 分片规则：哈希取模 / 范围分片
- 必须：**分片键做主键第一部分**
- 禁止：跨分片事务、跨分片批量查询

### 4. 事务优化
- 事务只包裹核心 SQL，禁止嵌套业务逻辑、HTTP 调用、Redis 操作
- 禁止分布式事务（高并发用最终一致性：消息队列、事务消息）
- 更新语句必须加 `WHERE` 条件，避免全表更新

### 5. 禁止的 SQL 写法（高并发严禁）
```sql
-- 1. 禁止 SELECT *（只查需要字段）
-- 2. 禁止 JOIN 超过 3 张表
-- 3. 禁止 IN 子查询（改用 JOIN 或 EXISTS）
-- 4. 禁止 ORDER BY RAND()
-- 5. 禁止大偏移分页 LIMIT 100000,20
```

## 六、反范式设计（高并发专用）
高并发场景**优先反范式，放弃严格第三范式**：
1. 允许字段冗余，减少 JOIN
2. 允许数据异步更新，追求最终一致性
3. 核心原则：**读多写少 → 冗余；写多读少 → 规范**

---

## 七、注释与文档规范
- 所有表必须加 **COMMENT** 说明业务含义
- 所有字段必须加 **COMMENT**，枚举值必须标注含义
- 索引必须说明用途
示例：
```sql
`pay_status` TINYINT NOT NULL DEFAULT 0 COMMENT '支付状态 0=待支付 1=支付成功 2=支付失败',
```

## 八、高并发表设计模板（可直接复制）
```sql
CREATE TABLE `业务模块_表名` (
  `id`          BIGINT UNSIGNED NOT NULL COMMENT '主键ID',
  `业务字段1`    VARCHAR(32) NOT NULL DEFAULT '' COMMENT '字段说明',
  `业务字段2`    BIGINT NOT NULL DEFAULT 0 COMMENT '字段说明',
  `业务状态`     TINYINT NOT NULL DEFAULT 0 COMMENT '状态 0=待处理 1=处理中 2=完成',
  `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `is_deleted`  TINYINT NOT NULL DEFAULT 0 COMMENT '删除标识 0=未删除 1=已删除',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_业务唯一字段` (`业务唯一字段`),
  KEY `idx_查询字段` (`查询字段`),
  KEY `idx_组合字段1_字段2` (`组合字段1`,`字段2`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='业务表说明';
```

---

### 总结
1. **高并发 MySQL 设计核心**：控制单表量、精简索引、禁止 NULL、拆分大字段、适度冗余
2. **死命令**：InnoDB + utf8mb4 + BIGINT 主键 + 逻辑删除 + 非空约束
3. **性能底线**：无长事务、无外键、无存储过程、无大表 JOIN
4. **扩展底线**：2000 万数据必须分表，高并发必须分库
