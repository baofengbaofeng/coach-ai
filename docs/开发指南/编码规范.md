# CoachAI项目编码规范

## 角色定义

你是 CoachAI 项目的 AI 开发助手。在生成或修改代码时，必须严格遵守以下规范。

## 语言规范（新增 2026-03-26）

### 代码注释语言（强制）
- **必须使用中文**：所有代码注释必须使用中文编写
- **文档字符串**：函数、类、模块的文档字符串必须使用中文
- **复杂逻辑**：复杂业务逻辑必须添加详细的中文注释
- **TODO注释**：TODO、FIXME等注释也必须使用中文

### 日志输出语言（强制）
- **必须使用英文**：所有日志输出必须使用英文
- **结构化日志**：日志消息和结构化字段都使用英文
- **日志级别**：DEBUG、INFO、WARNING、ERROR、CRITICAL日志都使用英文
- **禁止中文日志**：代码中不允许出现中文的日志输出

### 异常消息语言（强制）
- **必须使用英文**：所有异常抛出消息必须使用英文
- **错误描述**：异常消息应该清晰描述错误原因
- **禁止中文异常**：代码中不允许出现中文的异常抛出消息
- **业务异常**：自定义业务异常的消息也必须使用英文

### 示例代码
```python
# ✅ 正确示例 - 中文注释 + 英文日志/异常
def create_user(email: str, password: str) -> User:
    """
    创建新用户
    
    参数:
        email: 用户邮箱地址
        password: 用户密码
        
    返回:
        User: 创建的用户对象
        
    异常:
        ValueError: 如果邮箱已存在或密码不符合要求
    """
    # 检查邮箱是否已存在
    existing_user = user_repository.get_by_email(email)
    if existing_user:
        # ❌ 错误：raise ValueError("邮箱已存在")
        # ✅ 正确：
        raise ValueError("Email already exists")
    
    # 创建用户逻辑
    user = User(email=email, password=password)
    user_repository.save(user)
    
    # ❌ 错误：logger.info("用户创建成功", email=email)
    # ✅ 正确：
    logger.info("User created successfully", email=email)
    
    return user

# ✅ 正确示例 - 结构化日志
logger.info("User login successful", 
           user_id=user.id, 
           ip=request_ip, 
           duration=login_duration)

# ✅ 正确示例 - 业务异常
class UserNotFoundException(Exception):
    """用户未找到异常"""
    def __init__(self, user_id: int):
        # ❌ 错误：super().__init__(f"用户ID {user_id} 不存在")
        # ✅ 正确：
        super().__init__(f"User with ID {user_id} not found")
        self.user_id = user_id
```

## 禁止事项

1. 禁止使用 Tab 缩进
2. 禁止行尾留有空格
3. 禁止使用 print() 调试
4. 禁止裸 except
5. 禁止循环导入
6. 禁止函数内部 import（避免循环导入除外）

8. 禁止魔法数字
9. 禁止动态 import
10.禁止元编程
11.禁止隐式全局

## Python 编码规范

### 项目结构（强制）
- 遵循单一职责原则，一个模块 / 函数只完成一个功能
- 强制分层架构，接口层 → 业务逻辑层 → 数据访问层，禁止跨层直接调用
- 公共工具代码抽离为独立工具包，禁止重复编写冗余代码

### 模块划分原则
- 单一职责: 每个模块只负责一个功能领域
- 依赖倒置: 高层模块不依赖低层模块
- 接口隔离: 使用抽象基类定义接口

### 导入顺序（强制）
必须按以下顺序分组，每组内部按字母序排列：

1. Python标准库
2. 第三方库
3. 项目内部（按层级从浅到深）

### 导入注释（强制）
每行只能导入一个依赖项，且必须添加注释，含注释整行字符长度不小于60个字符，不多于120个字符。

### 命名规范（强制）
- 模块/包: 小写 + 下划线 (executor_factory.py)
- 类: 大驼峰 (ExecutorContext, BrowserSession)
- 函数/方法: 小写 + 下划线 (task_submit_handle())
- 常量: 大写 + 下划线 (MAX_RETRY_COUNT)
- 私有成员: 单下划线前缀 (_context_pool)
- 类型变量: 大驼峰 (T, ContextType)
- 所有命名都尽量避免使用缩写或者短命名，避免缩写二义性

### 类型注解（强制）
- 函数必须标注参数类型和返回类型
- 可选类型使用 | 语法 (Python 3.12+)
- 异步函数必须标注返回类型

### 文档字符串（强制）
- 必须包含 Args、Returns、Raises 部分，注释首行长度不得少于60个字符：

### 数据类定义（强制）
- 数据类注释保持在一行，整行字符长度不小于60个字符，不多于120个字符。
- 属性字段采用行内注释，整行字符长度不小于60个字符，不多于120个字符。

### 函数定义（强制）
- 函数定义必须包含参数列表和返回值注释，且首行注释长度不得少于60个字符，不多于120个字符。
- 函数参数只在参数列表字符长度超过120个字符时才进行换行。
- 尽量避免一行一个参数，换行后的参数对齐方式为普通缩进，不要使用参数对齐。

### 异常处理（强制）
- 禁止使用裸 except（except:），必须捕获具体异常类型
- 禁止空异常捕获（吞掉异常），必须记录日志或显式抛出
- 自定义业务异常类，区分系统异常与业务异常
- 文件、数据库、网络等资源操作必须使用 with 语句自动释放资源
- 使用异常链保留原始错误: `raise ... from e`
- 使用 try-finally 确保资源释放

### 日志规范（强制）
- 模块顶部定义: `_LOGGER = logging.getLogger(__name__)`
- 禁止使用 print() 调试
- 使用正确的日志级别

### 安全开发（强制）
- 绝对禁止硬编码敏感信息（密码、密钥、Token、数据库地址），必须通过环境变量或者配置文件注入
- 所有用户输入必须做参数校验 + 转义，防御 SQL 注入、XSS、命令注入、路径遍历
- 禁止使用高危函数：eval()/exec()/os.system() 处理不可信输入
- 定期扫描第三方依赖漏洞，禁止使用存在高危漏洞的包
- 生产环境禁用 DEBUG 模式，关闭错误堆栈暴露

### 性能优化（强制）
- 禁止在循环内执行 IO / 数据库操作，必须使用批量处理
- 大数据量处理优先使用生成器（yield），避免全量加载导致内存溢出
- 数据库操作必须使用索引，禁止全表扫描；复用连接池
- 性能瓶颈必须通过工具分析，禁止盲目优化

### 单元测试（强制）
- 所有的代码变更都需要检查是否存在对应的单元测试代码，如果存在则需要同步变更单测代码
